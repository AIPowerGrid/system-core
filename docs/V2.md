# system-core v2 — Architecture

**Status:** approved direction (2026-06-11). Nobody is on the legacy surface, so v2 is a
clean consolidation, not a compatibility exercise.

## The one-sentence version

Promote `grid_api` (FastAPI) to be the entire system, port the four things the Flask
horde still does (media queue, stats, registration, worker key checks), and delete the
Haidra layer — with wallet identity and decentralization seams designed in from the
start.

## Why now

An audit of every real client found the Flask horde's living surface is ~14 endpoints:

| Client | Endpoints actually used |
|---|---|
| comfy-bridge (media worker) | `/v2/generate/pop`, `submit`, `progress`, `preview/{id}` |
| grid-inference-worker (poll mode only — WS mode already uses grid_api) | `/v2/generate/text/pop`, `submit`, `/v2/find_user` |
| grid-frontend (dashboard) | `generate/async|check|status`, `generate/text/async|status`, `stats/{text,img}/{totals,models}`, `status/models`, `workers` |
| grid_api itself | proxies image jobs to Flask (`images.py`) |

Everything else — teams, shared keys, ratings, interrogation, kudos transfers, filters,
news — has **zero clients** and is deleted, not ported.

## Stack (what stays, what goes)

| | Decision |
|---|---|
| **API** | FastAPI + uvicorn, async end-to-end. One app. (Flask, waitress, the 8-proc pool: gone) |
| **DB** | Postgres primary, but the schema is **dialect-portable** (SQLAlchemy portable types, verified on SQLite) so a community gateway is one binary + an embedded DB. **Alembic** versioned migrations replace ad-hoc `init_database` and the ~40 Haidra tables. |
| **Storage model** | **Ledger-first**: the chain holds consensus-critical truth (registry, settlement roots, payments); off-chain state is either an append-only ledger (Merkle-izable, auditable against on-chain roots) or a rebuildable cache. See `grid_api/v2/schema.py`. |
| **Queue/streams** | Redis (Streams + pub/sub), as grid_api already does. No new broker. |
| **Background work** | Plain asyncio tasks in-app; **pg_cron and stored procedures are gone.** |
| **Observability** | prometheus-client (`/metrics`), structured logging. |
| **Units** | **den** is the meter, **AIPG** is the money. The word "kudos" does not exist in v2. |

Boring, proven, already half-deployed. The innovation budget is spent on the protocol
and the web3 seams, not on exotic infrastructure. ("Decentralized" is achieved by
minimal on-chain truth + verifiable logs + gateways anyone can run — not by an exotic
database.)

## Identity: wallet-native

A v2 **account** is one identity with up to three credentials:

- **wallet** (SIWE signature — same flow as aipg.chat and the art gallery; canonical id
  when present)
- **oauth** (Google/GitHub/Discord, via the dashboard)
- **api keys** (SHA-256 + GRID_SALT) — *derived* credentials of an account, not the
  account itself; multiple keys per account, revocable individually

Workers register over WS by signing with their wallet; den events, settlement Merkle
leaves, and PaymentRouter payouts all key on the same address. One identity across
chat, API, workers, gallery, rewards.

## Unified worker protocol (WS)

One WebSocket protocol for every worker type — text (exists today), image, video:

```
worker → register   {wallet_sig, models[], capabilities{type: text|image|video, ...}}
server → job        {id, type, payload, deadline, upload?: presigned_put_url[]}
worker → progress   {id, pct, preview_b64?}        (media)
worker → token      {id, delta}                    (text streaming)
worker → done       {id, receipt{prompt_hash, result_hash}, outputs[]}
server → ack        {id, den}
```

- Media results go **directly to R2 via presigned PUT URLs** — workers never hold
  storage credentials.
- Every `done` carries a **job receipt** (prompt hash + result hash), persisted from day
  one. That is the hook for future verifiable inference: receipts can later be attested,
  sampled, or challenged without redesigning the data path.
- Poll endpoints are not ported. comfy-bridge gets a WS client (mirroring
  grid-inference-worker's `ws_client.py`).

## Schema (fresh DB, clean slate — `grid_` namespace)

```
grid_accounts   id, wallet (canonical id), email, oauth_sub, flags
grid_api_keys   hash → account; many per account, individually revocable
grid_workers    wallet-keyed registry + presence; counters derivable from ledger
grid_jobs       CACHE — hot dispatch state, prunable after completion
grid_ledger     TRUTH — append-only job-completion events: den + prompt/result
                hashes (+ reserved worker_sig). Merkle root per epoch → chain.
grid_epochs     settlement periods: total_den, merkle_root, aipg_paid, tx_hash
```

Six tables instead of forty; receipts and den are **one ledger**, not two systems.
Alembic migration 0001 is the genesis (verified up/down on SQLite and Postgres
dialects). Legacy `grid_den_events` is absorbed into `grid_ledger` during the
media-port step.

## Decentralization seams (built in, activated later)

1. **Transport abstraction** — job dispatch goes behind an interface. Implementation #1
   is the WS broker; implementation #2 is libp2p gossipsub (worker already has
   `P2P_ENABLED` + bootstrap-peer config). The broker is a transport, not the
   architecture.
2. **Stateless gateway tier** — grid_api keeps all state in Postgres/Redis/chain, so the
   API can become N federated gateways anyone runs. (FLASK_API_BASE-style separation
   already proved the tier splits cleanly.)
3. **On-chain anchors (already live or built)** — model registry on the Grid Diamond
   (Base), den settlement Merkle roots (RewardPool/DenReporter, tested), PaymentRouter
   for AIPG payments. v2 writes the `settlements` row and the root in one motion.
4. **Receipts** (above) — the attestation hook.

Realistic path: gateways federate first; consensus-critical state (registry, settlement,
payments) is already on Base; dispatch goes p2p last.

### Implemented WS media protocol (v1)

```
server → job       {type:"job", id, job_type:"image"|"video", model, payload,
                    upload:[{put_url, key, content_type}]}
worker → progress  {type:"progress", id, pct, preview_b64?}
worker → done      {type:"done", id, results:[{index, seed, sha256}]}
server → ack       {type:"ack", id, den}
```

Worker uploads each output to its `put_url` (PUT, exact `content_type`), then reports
per-output `sha256` — that digest goes into the ledger receipt and is verifiable later
against the R2 object. Media den = megapixels × steps × n (× frames for video),
server-computed from the payload only (`den.py:calculate_media_den`).
